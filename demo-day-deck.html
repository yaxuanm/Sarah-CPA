User story 1

From competitor export to usable client portfolio.

The CPA uploads a CSV. DueDateHQ maps columns, flags ambiguous fields, separates new clients from updates, and derives deadline work from profile data.

CSVReal uploaded file, not hand-entered client cards.
AIUsed for messy column mapping and row interpretation.
NewNew clients become directory cards.
UpdateExisting clients are updated in place.
Open Clients -> Import -> upload CSV -> show mapping -> apply -> return to Client directory.

User story 2

Monday morning triage: what should the CPA do now?

Work nowActionable items where the CPA can move work today.
BlockedMissing documents, confirmation, or profile data.
Needs reviewExternal changes that require a CPA decision.
OverdueDue date has passed and the item is not archived.
ArchiveHandled work kept out of active triage.
Open Work -> filter if needed -> open one item -> edit task, file/revoke extension, draft client follow-up.

User story 3

A state tax change becomes a portfolio decision.

DueDateHQ watches official sources, interprets the update, shows before/after impact, and keeps the CPA as the final approver.

Source

Official source link and extracted rule change remain visible for review.

Impact

The product shows which clients are affected and exactly what changes.

Decision

The CPA can apply, dismiss, inspect affected-client changes, and see the result downstream.

Open Review -> source -> details -> Apply -> jump to changed client/work item evidence.

AI stance

AI is not decoration. It appears when messy input needs structure.

Where AI helps

Import mapping, row interpretation, official-source interpretation, affected-client summaries, and follow-up email drafts.

Where AI stops

It does not silently change tax records or submit professional work. Material changes stay behind CPA review and visible source evidence.

If asked: provider-backed AI runs when keys are configured; deterministic fallback keeps the demo stable.

On-demand rendering handoff

Adaptive interface, bounded by workflow.

User asks in natural language. The agent infers the decision need, not just keywords.

Backend gathers trusted data and emits a constrained render spec or known work surface.

Frontend renders the smallest useful surface instead of dumping another static page.

user intent -> inferred need -> data plan -> validated render spec -> actionable surface
