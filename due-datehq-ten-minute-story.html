Chapter 1

Sarah 不是在做税务工作。她在维护“什么时候该做什么”。

60个客户，Sarah 和一个兼职助理管理全部事项。
23个跨州经营客户，规则和截止日组合快速膨胀。
400+个联邦和州层面的截止日期需要持续追踪。
12h每周花在合规追踪和日历维护上。

这 12 小时不是客户建议、税务判断或收入工作，而是维护工作本身。

Current tools

她的工作流被拆在四个地方。

Excel

手工录入、手工更新，状态和规则变化都靠人记。

Calendar

能提醒日期，但看不到这件事是否已完成、被阻塞或不适用。

Email

客户沟通散落在线程里，催过没有、等什么文件，都不进入工作面。

微信群

偶尔从同行那里知道 IRS 或州税局发了新公告。

Real cost

48 小时的信息差，差点变成客户罚款和职业风险。

1整个周末被用来重新查状态。

Chapter 3

DueDateHQ 把认知负担从 Sarah 身上拿走，放进系统里。

1

规则库自动同步

联邦和州规则变更被监控、解析、匹配到客户，不再靠 Sarah 盯新闻。

2

优先级自动计算

不是按日期排序，而是综合紧急程度、blocked 状态和不可逆风险。

3

操作闭环

每次操作完成后，系统同步状态、规划下一步、给出入口。

Why AI

传统软件能存数据和发提醒，但不能接管判断链路。

理解短句“我已经发给对方了，记一下”不是闲聊，是记录一次客户催文件动作。
解析公告IRS 公告发布后，系统理解它影响哪个州、哪类客户、哪个截止日。
主动推进一件事完成后，系统判断同客户还有没有未处理事项，今天还有没有更紧急事项。

Chapter 4

这里不是一个飞轮，是三个独立系统共享同一个 CPA 用户基础。

1

规则库引擎

LLM 批量处理 IRS 和州税局公告。一次调用，影响所有用户的所有相关 deadline。

2

对话引擎

LLM 单次理解用户意图。飞轮不是扩大杠杆，而是让 Intent Cache 吃掉高频请求。

3

传统 SaaS

确定性管理客户、deadline、状态机、提醒和审计日志。这是数据底座和留存飞轮。

三个系统的飞轮逻辑完全不同，但都依赖同一个规模化 CPA 用户基础。

System 1

规则库引擎：LLM 一次调用，价值是 1:N。

公告IRS 或州税局发布非结构化公告。
解析LLM 一次调用，把公告变成结构化规则变更。
匹配系统匹配所有用户的所有受影响客户。
更新批量更新 deadline 和提醒。
验证用户纠正和确认行为反哺规则准确率。
100用户时，一次解析可能影响 500 个 deadline。
10000用户时，同样一次解析可能影响 50000 个 deadline。

System 1 flywheel

规则库引擎的护城河是被真实用户行为验证过的规则质量。

用户越多同一条规则变更影响的 deadline 越多。
验证越多有没有用户纠正、确认、忽略这条规则，都会成为信号。
准确率越高规则库质量不是纯人工审核，而是实际行为验证。
信任越强用户更愿意续费，更多用户进入规则库网络。

竞品可以抓公告，但拿不到“这条规则在真实 CPA 客户组合里是否被纠正过”的行为数据。

System 2

对话引擎：LLM 是 1:1 成本，所以飞轮是缓存收敛。

用户输入“发了”“标记完成”“下一个看谁”。
Intent Cache先查高频意图，命中就直接执行。
LLM未命中的长尾请求才调用模型理解。
执行打开工作面、记录操作、生成草稿、更新状态。
Library结果写回意图库，下次同类输入直接命中。

这里的目标不是让一次 LLM 调用影响更多用户，而是让越来越多请求不再需要 LLM。

System 2 flywheel

对话引擎能转起来，因为 CPA 的需求天然趋同。

输入越多更多 CPA 触发更多表达方式和任务模式。
覆盖越全Intent Library 覆盖高频、中频和部分低频需求。
命中率越高LLM 调用比例从 70-90% 降到 5-15%。
成本越低响应更快、毛利更高，也能用更低价格获客。
285条测试输入。
11个意图簇。
94%第二轮命中率，零错误匹配。

Long tail

“罕见需求”会被吞噬成“已知需求”。

早期：长尾很长

高频 ████████ 中频 ████ 低频 ██ 长尾 ░░░░░░░░░░░░░░░ 走 LLM

成熟期：长尾变短

高频 ████████ 中频 ████████ 低频 ██████ 长尾 ░░ 极少边界案例

不是 CPA 改变了，而是足够多用户触发过这些需求，系统已经学会。

Runtime staging

长尾需求第一次出现时，系统先当场满足，再决定是否沉淀。

新需求Sarah 提出一个没有命中缓存的复杂问题。
识别系统判断它不是已有 intent，也不是已有页面骨架。
临时工作面LLM 作为总设计师，用现有组件或临时组件搭出页面。
当场使用左侧立刻呈现，Sarah 可以查看分析、补字段、发起确认动作。
后台评估系统再判断这个场景是否值得保存为可复用资产。

Staging 不是让用户等待审核。它是当前会话立即可用、但尚未进入全局组件库的临时工作面。

Long-tail asset strategy

长尾不是都缓存。先存下来，再判断哪些值得资产化。

1

基础存储

所有长尾需求都记录：原始提问、上下文、数据来源、生成页面、用户操作和纠错信号。

2

缓存标准

只有高价值、可复用、低歧义、跨用户重复出现的场景，才进入 Intent Cache 或骨架库。

3

综合判断

系统判断它应该变成 intent、workflow skeleton、surface skeleton、draft pattern，还是继续保持长尾。

DueDateHQ 的数据资产不是“用户问过什么”，而是“我们知道一个 CPA 提问背后对应什么业务场景，以及这个场景是否值得产品化”。

System 3

传统 SaaS：没有 LLM，确定性管理业务数据。

客户录入客户、实体类型、州、联系人和税种进入系统。
Deadline根据规则和客户画像自动生成截止日。
状态机complete、snooze、waive、override 都有审计记录。
提醒截止日前触发提醒，完成或 waive 后取消。
留存数据越完整，迁移成本越高，续费越稳定。

这是传统 SaaS 的标准留存飞轮，也是系统一和系统二能规模化的用户底座。

Relationship

系统三是底座，系统一和系统二在底座上各自转动，再反哺底座。

系统三

提供稳定用户基础和业务数据。客户、deadline、状态和审计日志都在这里。

系统一

用同一个用户基础扩大 LLM 规则解析杠杆，规则越准，系统三的数据越准。

系统二

用同一个用户基础收敛 Intent Library，对话越准，系统三里的操作越高效。

传统 SaaS 提供数据和留存 -> 规则库引擎提升 deadline 准确率 -> 对话引擎提升操作效率 -> 两者都增加系统三的依赖度

Chapter 5

现在的产品：三个系统已经在同一个原型里合流。

传统 SaaS

规则录入、客户管理、deadline 调度、提醒推送、状态机和审计日志已经跑通。

规则库引擎

官方规则和客户组合可以生成受影响 deadline，支持人工验证和后续自动化。

对话引擎

285 条样本、11 个意图类型、Intent Cache 验证完成，Ask 已接入 live backend。

现场产品：Work 是全局掌控，Ask 是深度执行，两边共享状态。

Chapter 6

三个系统的成本下降速度不同。

系统成本机制下降逻辑
传统 SaaS基础设施和数据管理。随规模缓慢下降，幅度有限。
规则库引擎LLM 解析固定成本 + 人工审核。一次解析分摊到更多用户，单用户成本线性下降。
对话引擎每次输入原本是 1:1 LLM 成本。Intent Cache 命中率提升，调用比例快速下降。

早期如果每个 Ask 都带完整上下文调高质量模型，单用户月成本可能是几美元到十几美元。商业逻辑成立的关键，是 CPA 需求趋同让缓存命中率持续上升，而不是一开始就假设调用成本很低。

Conversation cost slope

对话成本一开始可能高于订阅价，之后随长尾被覆盖持续滑坡。

这不是“AI 便宜”的故事，而是“标准化职业的长尾需求会被逐步产品化”的故事。最终仍然有边界案例需要 LLM，所以成本曲线有尾巴，不会到零。

Scaling economics

规模越大，三个系统同时更有效率。

系统100 用户10000 用户
传统 SaaS基础设施约 $0.5-1 / 用户 / 月。约 $0.1-0.2 / 用户 / 月。
规则库引擎规则维护约 $2 / 用户 / 月。约 $0.02 / 用户 / 月。
对话引擎早期重度用户可能 $60-120 / 用户 / 月，轻中度用户约 $6-18。成熟后高频命中缓存，约 $2-6 / 用户 / 月，仍保留长尾。

系统一用 LLM 处理市场信息，杠杆是 1:N；系统二用 LLM 理解用户意图，早期成本高，飞轮是缓存收敛；系统三用确定性逻辑管理业务数据，飞轮是数据留存。

Close

DueDateHQ 保护的，是 CPA 的判断力。

一个 CPA 最有价值的东西，从来不是她记住了多少截止日期。是她对客户情况的理解，和她给出建议时的判断力。

规则库更新，不需要她盯着 IRS 官网。优先级排序，不需要她每天早上想 25 分钟。操作记录，不需要她手动更新 Excel。

这些事情系统做了，Sarah 的时间和注意力就留给真正需要她的事：判断、沟通、建议。
